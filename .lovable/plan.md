

# Security Audit and Branding Cleanup

## Overview
Rename all user-facing "lovable" references in the AI provider system to "wiseresume", clean up rate limiter naming, update the Play Store URL placeholder, and address the security concern in `safeClient.ts`. Several items from the uploaded plan **cannot be implemented** due to platform constraints (detailed below).

## Items That CANNOT Be Changed

| Item | Reason |
|------|--------|
| `src/integrations/lovable/index.ts` | Auto-generated by the platform. Must not be deleted or modified. |
| `vite.config.ts` lovable-tagger | Platform infrastructure for the development environment. Must stay. |
| `@lovable.dev/cloud-auth-js` package | Required for Google/Apple OAuth on this platform. Removing it breaks social sign-in. |
| `AuthPage.tsx` OAuth replacement | The `lovable.auth.signInWithOAuth()` call is the correct way to do OAuth on this platform. Replacing it with direct Supabase OAuth would break Google/Apple login. |
| Edge function `LOVABLE_API_KEY` env var | This is a platform-provisioned secret. Renaming it requires updating the secret on the backend, and the current name is the one the platform expects. Renaming risks breaking all AI features. |
| `.env` file | Auto-generated and managed by the platform. |

## Technical Details

### Zustand Migration
The `settingsStore` uses `persist` with key `wiseresume-settings`. Existing users who have `aiProvider: 'lovable'` stored in localStorage will get the Zustand default (`'wiseresume'`) since `'lovable'` won't match the new type. This is the desired behavior -- they stay on the default AI provider.

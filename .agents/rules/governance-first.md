# Governance First

Always read `project-governance/CONSTITUTION.md` and the relevant files in `project-governance/` before planning, specifying, or implementing any change.

Treat `project-governance/` as the primary source of truth for this repository.

If governance files conflict with legacy docs, historical notes, rebuild prompts, or old folders such as `enhancements-for-vibe-coding/`, governance files win.

Before making any change:
1. Sync with the latest repository state.
2. Review the relevant governance files.
3. Inspect the affected code and current implementation.
4. Propose the best approach in simple language.
5. Ask before any high-impact or risky change.

Do not assume outdated branding, auth setup, deployment setup, or architecture details are correct without checking the current codebase and governance files first.

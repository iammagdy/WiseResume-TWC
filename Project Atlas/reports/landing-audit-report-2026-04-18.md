# Landing-page Audit — 2026-04-18

Investigation focus:

1. Why does the app feel slow to start?
2. Are there real (not cosmetic) UI/UX bugs on the landing page and the
   scroll-stack section?

All findings are reproducible from the screenshots in `screenshots/audit-*.jpg`
(captured at 1280×800 desktop and 412×915 mobile, dark + light) and the boot
profile captured against the live dev server (Vite 6 + Express 5001, headless

## 1. Boot / "long startup" investigation

### Measured numbers

| Metric                          | Value      | Verdict |
|---------------------------------|------------|---------|
| Vite dev server ready           | **379 ms** | Fast |
| Express API ready (parallel)    | < 1 s      | Fast |
| `npm install --legacy-peer-deps` (every workflow start) | **~2 s** ("up to date, audited 1238 packages") | Avoidable |
| TTFB (first request)            | 47 ms      | Excellent |
| DOMContentLoaded                | 683 ms     | Good |
| Load event                      | 690 ms     | Good |
| First Paint                     | 1 649 ms   | Mediocre |
| **First Contentful Paint**      | **2 792 ms** | **Poor** (Web Vitals threshold: <1 800 ms = good, <3 000 ms = needs improvement) |
| Network idle                    | **17.6 s** | Poor (lazy chunks keep arriving) |
| Total resource requests         | **250**    | Very high for a single landing route |

### Root causes (real, not perceived)

**B-1 · Vite serves every component as a separate module in dev mode.**
The 250-request number isn't a config bug — it's normal Vite dev behaviour —
but it explains the "long start" feeling. Each landing dependency (~300 ms
average, e.g. `PageSkeletons.tsx` 355 ms, `BottomSheetContext.tsx` 330 ms,
`useWiseHireAccount.ts` 325 ms) is its own HTTP round-trip. Production builds
collapse these into 2–3 chunks, so the feel will improve dramatically once
deployed.

**B-2 · Google Fonts CSS is render-blocking and the slowest single resource
(1 557 ms).** `link rel="stylesheet" href="…fonts.googleapis.com/css2…Inter…"`
sits in `<head>` with no `font-display: swap` preconnect and no `<link
rel="preconnect">` to `fonts.gstatic.com`. This is the single biggest FCP
contributor.
*Fix*: add
```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
```
to `index.html`, and self-host Inter via `@fontsource/inter` to remove the
3rd-party round-trip entirely.

**B-3 · Eager `framer-motion` import in `src/pages/Index.tsx` line 8.**
`import { useReducedMotion, motion, AnimatePresence } from 'framer-motion'` is
top-level — it ships ~80 kB of motion code in the initial chunk even though
most of it is only used by lazy-loaded children. Move the heavy bits behind
the same `lazy()` boundary as `WiseResumeContent` / `WiseHireContent`.

**B-4 · `npm install --legacy-peer-deps` runs on every workflow start.**
Even when there are no changes, that's ~2 s wasted. The Replit workflow
should skip install when `node_modules` is up-to-date (`npm ci` against a
hash, or simply `[ -d node_modules ] || npm install …`).

**B-5 · Two Supabase clients are instantiated at boot. (REAL bug — not a
warning to silence.)** Browser console shows:
```
GoTrueClient @ sb-jnsfmkzgxsviuthaqlyy-auth-token (2.103.0)
Multiple GoTrueClient instances detected in the same browser context …
```
`src/integrations/supabase/client.ts` (4 callers — `useBulkScreen`,
`useScorecards`, `useWaitlist`, `ScorecardPage`) and
`src/integrations/supabase/safeClient.ts` (89 callers, the canonical bridge
client) both call `createClient()` against the same project, so the auth
storage key collides. Effects:
* duplicated auth listeners → wasted memory & extra wake-ups,
* the four legacy callers bypass the Kinde→Supabase JWT bridge and rely on
  `sessionStorage` auth that is never populated in the bridge flow → those
  hooks will silently fail to authorise once the app moves off the legacy
  Supabase Auth path.

*Fix*: migrate the 4 stragglers to `safeClient.ts` and delete the legacy
`client.ts` (the comment "This file is automatically generated. Do not edit
it directly." is misleading — nothing regenerates it now).

**B-6 · WebGL "GPU stall due to ReadPixels" warnings at boot (4×).** Repeated
warnings from `[.WebGL-…]GL Driver Message` indicate a backdrop-filter
composited layer is forcing the GPU to read back pixels per frame. The
landing page has at least 4 `backdrop-filter: blur(…)` rules
(`index-landing.css` lines 144, 689, 706 and `LandingHeader.tsx`) plus a
`filter: blur(40px)` ornamental glow. Stacking these on the same compositor
layer causes the stalls. Low priority unless the user reports jank on weak
GPUs, but worth tracking — disable backdrop-filter on `prefers-reduced-motion`
or below a viewport-width threshold.

**B-7 · Tailwind warning on every HMR**: `The class duration-[1200ms] is
ambiguous and matches multiple utilities`. Cosmetic but pollutes the dev log.
Replace with `duration-1000` or wrap as `[transition-duration:1200ms]`.

## 2. UI/UX issues on the landing page

All issues confirmed against captured screenshots (16 shots: 4 scroll
positions × 2 schemes × 2 viewports). Marked **REAL** vs. **EXPECTED** so the
checklist is unambiguous.

### REAL bugs

**U-1 · Scroll-stack background "ghost UI" leaking from prior cards.**
`screenshots/audit-desk-dark-mid-stack.jpg` and `audit-desk-dark-late-stack.jpg`
show stray UI fragments floating above the active card:
* a rounded grey pill near the top of the viewport (mid-stack shot),
* an "Injecting keywords…" pill plus a small red icon (late-stack shot).

These are inner UI elements from the **previous** card's mock that escape the
card's `overflow: hidden` because they live inside `.lp-stack-parallax`,
whose own `transform: translateY(...)` pushes them visually outside the
card boundary while their offset parent is still pinned. Since the
yesterday's fix made cards opaque, those "escaped" parallax elements now
render on the page background instead of being hidden by the next card —
they were always escaping, the previous transparent cards just hid the
problem.

*Fix candidates*:
- Add `clip-path: inset(0 round var(--card-radius))` to
  `.lp-stack-section .scroll-stack-card`, **or**
- Cap the parallax `translateY` so it cannot drag content past the card
  edges, **or**
- Wrap each parallax child in an inner element with `overflow: hidden`.

**U-2 · Mock-phone overflow on mobile mid-stack.** In
`audit-mob-light-mid-stack.jpg` the "Smart Tailoring" phone mock visibly
extends below its card's bottom border into the next card's area. Same root
cause as U-1 (`.lp-stack-parallax` shifts content past the card edge). On
mobile this is worse because the card is the same width as the viewport, so
the overflow is impossible to ignore.

**U-3 · Watermark text leaks visually behind cards.** The huge faded
"Tailoring" / "Interview" watermark is intended decoration, but in dark
mode it bleeds outside the card frame onto the section background (visible
in `audit-desk-dark-mid-stack.jpg` around y=380). Either move the watermark
inside `.scroll-stack-card { overflow: hidden }` or reduce its scale by 10–15 %
so it stops escaping at narrow viewports.

**U-4 · Footer link contrast (light mode).** In
`audit-desk-light-bottom.jpg` the "Privacy Policy | Terms of Service" links
render in a light pink/grey on a pink background. Eyeballed contrast ratio
≈ 1.6 : 1, far below WCAG AA's 4.5 : 1. The "Your data is encrypted and
secure" caption has the same issue.

**U-5 · "Sign In" + active toggle pill use the same red.** The pill
"Individuals" (active) and the "Sign In" button (`audit-desk-*-top.jpg`) are
both filled in `--lp-brand` red. From a quick glance it's unclear which is
the primary CTA. Pick one — typically the toggle would be a softer state
(border + tint) so the CTA pops.

**U-6 · `[server]` warning during sweep**: `[analytics-sweep] skipped — lock
row held by another holder`. Not user-visible, but it indicates the sweep
locks aren't being released between hot-restarts. Worth a follow-up to
clear stale lock rows on dev startup.

### EXPECTED (verified, not bugs)

* "Senior Develope" mid-typewriter cursor in `audit-desk-dark-top.jpg` —
  this is the rotating-word animation captured mid-tween, not a truncation.
* Blank ~30 vh spaces between cards — that's the intended runway for the
  pin-and-stack scroll choreography.
* Light-mode parallax watermark is barely visible — by design (low opacity
  on bright background).

## 4. What "long startup" actually is

* The **dev workflow** is fast (Vite 379 ms + API < 1 s).
* The **first paint** is poor because of (a) 1.5 s blocked on Google Fonts
  CSS and (b) eager framer-motion in the initial chunk.
* The **"network never settles"** feeling is dev-mode-only — 250 separate
  module requests. Production build will collapse these.
* Two parallel auth clients add ~200–400 ms of duplicate init work and
  produce the multi-instance warning.

In short: in dev the perceived slowness is dominated by Google Fonts and
the eager framer-motion import; in production it should drop to the 1.6–1.8 s
FCP range once B-2 and B-3 are addressed.

# /act-as — Impersonation Landing

**Last verified:** 2026-05-08
**Type:** page reference card
**Sources:** `src/pages/ActAs.tsx`, `src/lib/impersonationStore.ts`, `src/App.tsx` (route definition)

## Route

`/act-as#<base64(JSON({t,u,e,x}))>`

Mounted at the App-level (NOT inside `AppInterior`) so it loads BEFORE the auth gate runs — credentials must be installed in `sessionStorage` before any auth-dependent route renders.

## What it does

Receives admin impersonation credentials in the URL **hash fragment** (never sent to any server). Parses, stores in `sessionStorage`, then navigates to `/dashboard`.

No server claim round-trip is needed, so this works in both dev and production.

## Hash payload schema

```
btoa(JSON.stringify({
  t: access_token,    // session token for the impersonated user
  u: user_id,         // bridge UUID
  e: email,
  x: expires_at,      // ms since epoch
}))
```

## Flow

1. `parseHashPayload()` — `atob` + `JSON.parse` + shape validation. Throws on missing fields.
2. `startImpersonation(payload)` — writes to `sessionStorage` (key managed by `impersonationStore.ts`).
3. `useNavigate()` → `/dashboard`.
4. On error: renders the `<UserX>` error card with the parse failure message (no automatic redirect).

## How impersonation tokens are issued

Generated by the `admin-impersonate` function (admin-auth gated). The DevKit "Act as" button on the user list constructs the `/act-as#…` URL and opens it in a new tab.

## Security notes

- Hash fragment never leaves the browser (no HTTP transmission).
- `sessionStorage` clears on tab close — impersonation is tab-scoped.
- Revocation: `admin-impersonate` writes to `impersonation_revocations`; the The issued token continues to work until natural expiry, but `useMe` checks `impersonation_revocations` and forces logout if revoked.

## Task #19 — Datadog LLM Observability added to ai-gateway

**Status:** MERGED ✅

### What was done

**`appwrite-hubs/ai-gateway/src/main.js`**
- Added `dd-trace` v5 import and tracer init at module level (`ddTrace.init({ logInjection: false })`).
- `enableLLMObs()` reads `DATADOG_API_KEY || DD_API_KEY` and `DD_SITE` from env; calls `tracer.llmobs.enable({ mlApp: 'wiseresumeai', agentlessEnabled: true })`. Agentless mode required — Appwrite Functions cannot run a Datadog agent sidecar. Silent no-op if key absent.
- `flushDD()` helper: calls `llmobs.flush()` + `tracer.flush()` (promise). Called before every `res.json()` return.
- AI route: when LLMObs enabled, `axios.post` is wrapped in `llmobs.trace({ kind: 'llm', name: featureName, modelName: model, modelProvider: provider })`. Annotates input messages, output content, feature/provider/model tags, and token usage metrics. On error: sets `error` + `error.message` span tags and re-throws so existing 500 path is preserved.
- Email routes (`send-email`, `send-contact-email`): NOT traced as LLM spans; `flushDD()` still called.

**`appwrite-hubs/ai-gateway/package.json`**
- Added `"dd-trace": "^5.102.0"`.

**`appwrite-hubs/ai-gateway.tar.gz`** — rebuilt (35 MB).

### Env vars required (set in Appwrite Console → Function Variables, NOT Replit)
- `DD_API_KEY` (or `DATADOG_API_KEY`) — Datadog API key
- `DD_SITE` — e.g. `datadoghq.com`

### Code review note
- APPROVED_WITH_COMMENTS: non-blocking. Notes: standardise on one API key var name in ops docs; add an alert for repeated "trace setup fallback" log events after deploy.

## Task #11 — NVIDIA NIM model dropdown updated to 5 Mistral/Gemma models

**Status:** MERGED ✅

### Root cause
The NVIDIA NIM slot in the DevKit AI Keys panel used a free-text input defaulting to `nvidia/llama-3.1-nemotron-70b-instruct` (retired from desired list). Both the frontend fallback constant and the `inspect-ai-keys` backend default referenced this old ID. The `<select>` added by this task would show blank for any slot carrying the old value.

### Fixes

**`src/lib/devkit/aiTestSlotModels.ts`**
- Exported `NVIDIA_LLM_MODELS: ReadonlyArray<{ label, value }>` — exactly 5 entries:
  - `mistral-medium-3-instruct` (new default)
  - `mistral-large-3-675b-instruct-2512`
  - `mistral-nemotron`
  - `gemma-3n-e4b-it`
  - `gemma-3n-e2b-it`
- `FALLBACK_AI_TEST_DEFAULT_MODELS.nvidia`: `'nvidia/llama-3.1-nemotron-70b-instruct'` → `'mistral-medium-3-instruct'`.

**`src/components/dev-kit/AIKeysPanel.tsx`**
- Replaced free-text `<input>` for NVIDIA slots with a `<select>` populated by `NVIDIA_LLM_MODELS`. Other providers (OpenRouter, Groq, DeepSeek) keep free-text inputs.
- `DEFAULT_MODELS.nvidia` updated to `'mistral-medium-3-instruct'`.
- Client-side normalization in `load()`: NVIDIA draft values not in `NVIDIA_LLM_MODELS` are coerced to `'mistral-medium-3-instruct'` — guards against stale DB overrides that survive backend normalization.

**`appwrite-hubs/inspect-ai-keys/src/main.js`**
- `DEFAULT_MODELS.nvidia` → `'mistral-medium-3-instruct'`.
- Added `NVIDIA_VALID_MODELS` array (same 5 IDs). At response-build time, any saved override not in this list is treated as `null` and falls back to the new default — prevents the panel from receiving an unrecognised model ID.

**`src/components/dev-kit/DevKitRunner.tsx`**
- `ai-engine-nvidia` label: `'Engine · NVIDIA (Nemotron 70B)'` → `'Engine · NVIDIA NIM (Mistral Medium 3)'`.
- Description: removes reference to `nvidia/llama-3.1-nemotron-70b-instruct`.

### Code review note
- APPROVED_WITH_COMMENTS: non-blocking. Raw `slotModels` returned to client still contains stale value if one is in the DB (affects "custom/default" badge only, not active model). Follow-up: validate NVIDIA model IDs on POST save in `inspect-ai-keys` to reject unsupported values server-side.

## Task #10 — Per-feature AI provider routing in ai-gateway

**Status:** MERGED ✅ (task agent)

### What was done

**`appwrite-hubs/ai-gateway/src/main.js`**
- Added `FEATURE_ROUTES` map: 22 `featureName → { provider, model }` entries covering all major AI features.
- `buildPool()` helper: extracts existing env-var pool construction.
- `pickProvider(featureName, pool)`:
  - If `featureName` has a `FEATURE_ROUTES` entry AND that provider has ≥1 configured key → uses preferred provider + model.
  - Otherwise → random pick from full pool (original fallback preserved).
- Response now includes `modelUsed` and `routedByFeature` fields.

|----------|----------|

| DeepSeek | analyze-resume, generate-fix-suggestions |

| Random pool | score-resume, ask-portfolio, coupons, others |

## Filename conventions

Two filename styles coexist in this project:

1. **UUID-named** (`<timestamp>_<uuid>.sql`) — generated by the Supabase CLI's `db diff` workflow. Topic must be inferred from the SQL body.
2. **Slug-named** (`<timestamp>_<descriptive_slug>.sql`) — handwritten, current convention. Topic is encoded in the filename.

All new migrations MUST use the slug-named style. Use `snake_case`, ≤60 chars, descriptive (e.g. `20260605000000_parse_job_ai_routing_config.sql`).

## Maintenance

When adding a new migration:

1. Use the slug-named form: `<YYYYMMDDHHMMSS>_<snake_case_topic>.sql`.
2. Include a top-of-file comment explaining the **why**, not just the **what**.
3. Add a row to the table above.
4. Update any affected reference cards in `Project Atlas/01-Currently Implemented/database-tables/`.
5. If touching RLS, also update `Project Atlas/01-Currently Implemented/critical-systems/09-security-model.md`.

## Hard rules (from `replit.md`)

- **Never change primary key column types** — destructive and breaks existing data.
- **Always additive** — drops and renames require explicit owner sign-off.
- All migrations must be idempotent (`IF NOT EXISTS`, `CREATE OR REPLACE`, etc.).
